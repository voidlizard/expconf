The algorithm/combinatorial_repeat.h header defines an algorithm that produces a combinatorial repetition of a dataset.

Usage

#include <chaos/preprocessor/algorithm/combinatorial_repeat.h>

Contents

  • CHAOS_PP_COMBINATORIAL_REPEAT

CHAOS_PP_COMBINATORIAL_REPEAT

The CHAOS_PP_COMBINATORIAL_REPEAT macro generates repetitions for all possible strings (in the mathematical sense) up to the length specified by its first argument of an alphabet specified by its third argument.

Usage - C99 Specific 

CHAOS_PP_EXPR(CHAOS_PP_COMBINATORIAL_REPEAT(count, macro, g))
count

The maximum length of the strings to be generated. Valid values range from 0 to CHAOS_PP_LIMIT_MAG.

macro

The target that is to be repeated for each string. This target must be either a λ expression or a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

macro(s, size, product)
g

The generic data structure containing the alphabet used to construct strings.

Usage - C99 Specific 

CHAOS_PP_EXPR(CHAOS_PP_COMBINATORIAL_REPEAT(count, macro, g, ...))
count

The maximum length of the strings to be generated. Valid values range from 0 to CHAOS_PP_LIMIT_MAG.

macro

The target that is to be repeated for each string. This target must be either a λ expression or a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

macro(s, size, product, data†)

† The data pseudo-parameter must be either ... or the number of parameters that corresponds to the number of arguments in the auxiliary data, __VA_ARGS__.

g

The generic data structure containing the alphabet used to construct strings.

[...]

Auxiliary data to be passed through the algorithm to the target.

Remarks

The CHAOS_PP_COMBINATORIAL_REPEAT macro itself expands to a deferred expression in terms of CHAOS_PP_NEXT(s). This deferred expression, when invoked with CHAOS_PP_EXPR, expands to the result of the repeated invocoation of the target for each string (passed as a sequence) in the alphabet specified by g up to a maximum length of count. For example, if the alphabet is {a,b} and the maximum length of string is three, the generated strings are:

{a} {b} {a,a} {a,b} {b,a} {b,b} {a,a,a} {a,a,b} {a,b,a} {a,b,b} {b,a,a} {b,a,b} {b,b,a} {b,b,b}

In other words, the macro is generating Cartesian products of g with dimensions ranging from 1 to count. 

#define M(s, size, product) \
    { CHAOS_PP_SEQ_ENUMERATE(product) } \
    /**/

CHAOS_PP_EXPR(CHAOS_PP_COMBINATORIAL_REPEAT(
    3, M, (CHAOS_PP_SEQ) (A)(B)
))

The target is invoked with the current recursion state (s), the size of a particular string (size), the particular string (product), and, if it exists, the auxiliary data (__VA_ARGS__). This target may expand to a deferred expression in terms of CHAOS_PP_NEXT(s). This allows the target to recurse on itself if necessary.

It is possible to omit the auxiliary data argument altogether. This is different than passing "nothing" as the auxiliary data. 

#define D (CHAOS_PP_TUPLE) (a, b, c)

CHAOS_PP_EXPR(
    CHAOS_PP_COMBINATORIAL_REPEAT(3, MACRO, D,) // #1
    CHAOS_PP_COMBINATORIAL_REPEAT(3, MACRO, D)  // #2
)

The first line passes "nothing" as the auxiliary data, which causes MACRO to be invoked as (e.g.)... 

MACRO(s, 3, (a)(a)(a),)

The second line omits the auxiliary data, which causes MACRO to be invoked as (e.g.)... 

MACRO(s, 3, (a)(a)(b))

The fundamental difference is that, in the first line, the auxiliary data is empty (which might have a valid meaning as data), but, in the second line, there is no auxiliary data. Hence, the algorithm is overloaded on a true-optional parameter.

Invocations of the target are trampolined such that each invocation is invoked with CHAOS_PP_NEXT(s), where s is the original recursion state at the beginning of the algorithm (i.e. CHAOS_PP_STATE()). This maximizes the utility of the recursion backend.

This macro deduces the current recursion state and invokes CHAOS_PP_COMBINATORIAL_REPEAT_S. If the current recursion state is already known, direct use of CHAOS_PP_COMBINATORIAL_REPEAT_S is more efficient.

This macro may not be used if bypass semantics are in effect.

Sample Code

#include <chaos/preprocessor/algorithm/combinatorial_repeat.h>
#include <chaos/preprocessor/config.h>
#include <chaos/preprocessor/facilities/unbox.h>
#include <chaos/preprocessor/recursion/expr.h>
#include <chaos/preprocessor/repetition/enum.h>
#include <chaos/preprocessor/repetition/enum_params.h>
#include <chaos/preprocessor/seq/core.h>
#include <chaos/preprocessor/seq/elem.h>

#if CHAOS_PP_VARIADICS

template<class Obj> class forward {
    public:
        // ...
        #define A(s, size, product) \
            template<CHAOS_PP_ENUM_PARAMS(size, class T)> \
            typename result_of<Obj(CHAOS_PP_ENUM(size, B, product))>::type \
            operator()(CHAOS_PP_ENUM(size, C, product)) { \
                return obj(CHAOS_PP_ENUM_PARAMS(size, p)); \
            } \
            /**/
        #define B(s, n, product) \
            CHAOS_PP_UNBOX(CHAOS_PP_SEQ_ELEM(n, product)) T ## n \
            /**/
        #define C(s, n, product) \
            B(s, n, product)& p ## n \
            /**/
        CHAOS_PP_EXPR(CHAOS_PP_COMBINATORIAL_REPEAT(
            5, A, (CHAOS_PP_SEQ) (.)((const))
        ))
        #undef A
        #undef B
        #undef C
    private:
        Obj obj;
};

#endif

CHAOS_PP_COMBINATORIAL_REPEAT_ID

The CHAOS_PP_COMBINATORIAL_REPEAT_ID macro indirectly expands to CHAOS_PP_COMBINATORIAL_REPEAT.

Usage - C99 Specific 

CHAOS_PP_COMBINATORIAL_REPEAT_ID()

Remarks

This macro effectively allows CHAOS_PP_COMBINATORIAL_REPEAT to pass through itself without becoming disabled.

CHAOS_PP_COMBINATORIAL_REPEAT_

The CHAOS_PP_COMBINATORIAL_REPEAT_ macro expands to a λ expression representing CHAOS_PP_COMBINATORIAL_REPEAT.

Usage - C99 Specific 

CHAOS_PP_COMBINATORIAL_REPEAT_

Remarks

This macro is a shorthand form of the following λ binding: 

CHAOS_PP_LAMBDA(CHAOS_PP_COMBINATORIAL_REPEAT_ID)()
