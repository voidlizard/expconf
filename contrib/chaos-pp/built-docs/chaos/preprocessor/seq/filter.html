The seq/filter.h header defines an algorithm that filters elements from a sequence according to a user-specified criteria.

Usage

#include <chaos/preprocessor/seq/filter.h>

Contents

  • CHAOS_PP_SEQ_FILTER

CHAOS_PP_SEQ_FILTER

The CHAOS_PP_SEQ_FILTER macro filters elements from its second argument according to the criteria specified by its first argument.

Usage - C99 Specific 

CHAOS_PP_EXPR(CHAOS_PP_SEQ_FILTER(pred, seq))
pred

The predicate that determines whether an element is to remain in the resulting sequence. This predicate must be either a λ expression or a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

pred(s, elem†)

† If seq contains elements that have varying arities, elem must be .... Otherwise, elem must be the number of parameters that corresponds to the arity of each element in seq.

[seq]

The sequence to be filtered.

Usage - C99 Specific 

CHAOS_PP_EXPR(CHAOS_PP_SEQ_FILTER(pred, seq, ...))
pred

The predicate that determines whether an element is to remain in the resulting sequence. This predicate must be either a λ expression or a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

pred(s, elem†, data‡)

† The elem pseudo-parameter must be the number of parameters that corresponds to the arity of each element in seq.

‡ The data pseudo-parameter must be either ... or the number of parameters that corresponds to the number of arguments in the auxiliary data, __VA_ARGS__.

[seq]

The sequence to be filtered.

[...]

Auxiliary data to be passed through the algorithm to the predicate.

Usage - C/C++ Specific 

CHAOS_PP_EXPR(CHAOS_PP_SEQ_FILTER(pred, seq, data))
pred

The predicate that determines whether an element is to remain in the resulting sequence. This predicate must be a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

pred(s, elem, data)
seq

The sequence to be filtered.

data

Auxiliary data to be passed through the algorithm to the predicate.

Remarks

The CHAOS_PP_SEQ_FILTER macro itself expands to a deferred expression in terms of CHAOS_PP_NEXT(s). This deferred expression, when invoked by CHAOS_PP_EXPR, expands to a sequence that contains the elements for which the predicate expands to 1.

The predicate is invoked for each element in seq with the current recursion state (s), the current element (elem), and, if it exists, the auxiliary data (data or __VA_ARGS__). This invocation must expand to 0, 1, or a deferred expression in terms of CHAOS_PP_NEXT(s) that expands to either 0 or 1. Expansion to a deferred expression allows the predicate to recurse on itself if necessary.

In C99, it is possible to omit the auxiliary data argument altogether. This is different than passing "nothing" as the auxiliary data. 

CHAOS_PP_EXPR(
    CHAOS_PP_SEQ_FILTER(PRED, (a)(b)(c),) // #1
    CHAOS_PP_SEQ_FILTER(PRED, (a)(b)(c))  // #2
)

The first line passes "nothing" as the auxiliary data, which causes PRED to be invoked as (e.g.)... 

PRED(s, a,)

The second line omits the auxiliary data, which causes PRED to be invoked as (e.g.)... 

PRED(s, a)

The fundamental difference is that, in the first line, the auxiliary data is empty (which might have a valid meaning as data), but, in the second line, there is no auxiliary data. Hence, the algorithm is overloaded on a true-optional parameter.

Invocations of the predicate are trampolined such that each invocation is invoked with CHAOS_PP_NEXT(s), where s is the original recursion state at the beginning of the algorithm (i.e. CHAOS_PP_STATE()). This maximizes the utility of the recursion backend.

This macro uses sequential iteration. As a result, the efficiency of the algorithm's use of the recursion backend, described in terms of the number of recursion steps required to complete the algorithm and ignoring the complexity of pred, is a constant Θ(1). As such, the algorithm is theoretically capable of processing sequences containing an unrestricted number of elements.

This macro deduces the current recursion state and invokes CHAOS_PP_SEQ_FILTER_S. If the current recursion state is already known, direct use of CHAOS_PP_SEQ_FILTER_S is more efficient.

This macro may not be used if bypass semantics are in effect.

Sample Code

#include <chaos/preprocessor/config.h>
#include <chaos/preprocessor/lambda/ops.h>
#include <chaos/preprocessor/logical/bool.h>
#include <chaos/preprocessor/recursion/expr.h>
#include <chaos/preprocessor/seq/filter.h>

#define PRED(s, elem, data) CHAOS_PP_BOOL(elem)

CHAOS_PP_EXPR(CHAOS_PP_SEQ_FILTER(
    PRED, (4)(6)(0)(0)(8)(0), ~
))
// (4)(6)(8)

#if CHAOS_PP_VARIADICS

CHAOS_PP_EXPR(CHAOS_PP_SEQ_FILTER(
    CHAOS_PP_BOOL_(CHAOS_PP_ARG(1)),
    (4)(6)(0)(0)(8)(0)
))
// (4)(6)(8)

#endif

CHAOS_PP_SEQ_FILTER_ID

The CHAOS_PP_SEQ_FILTER_ID macro indirectly expands to CHAOS_PP_SEQ_FILTER.

Usage 

CHAOS_PP_SEQ_FILTER_ID()

Remarks

This macro effectively allows CHAOS_PP_SEQ_FILTER to pass through itself without becoming disabled.

CHAOS_PP_SEQ_FILTER_

Usage - C99 Specific 

CHAOS_PP_SEQ_FILTER_

Remarks

This macro is a shorthand form of the following λ binding: 

CHAOS_PP_LAMBDA(CHAOS_PP_SEQ_FILTER_ID)()
