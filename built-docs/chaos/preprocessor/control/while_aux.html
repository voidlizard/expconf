The control/while_aux.h header defines a looping construct.

Usage

#include <chaos/preprocessor/control/while_aux.h>

Contents

CHAOS_PP_WHILE_AUX

The CHAOS_PP_WHILE_AUX macro iterates a state with a user-specified operation while a user-specified predicate expands to 1.

Usage - C99 Specific 

CHAOS_PP_EXPR(CHAOS_PP_WHILE_AUX(pred, op, data, ...))
pred

The predicate that determines whether the loop should continue. This predicate must be either a λ expression or a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

pred(s, data†, state‡)

† If the auxiliary data is parenthetic, the data pseudo-parameter must be the number of parameters that corresponds to the number of sub-elements in the auxiliary data. Otherwise, it must be a single parameter.

‡ The state pseudo-parameter must be either ... or the number of parameters that corresponds to the number of arguments in the initial state, __VA_ARGS__.

op

The operation used to iterate the state. This operation must be either a λ expression or a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

op(s, data†, state‡)

† If the auxiliary data is parenthetic, the data pseudo-parameter must be the number of parameters that corresponds to the number of sub-elements in the auxiliary data. Otherwise, it must be a single parameter.

‡ The state pseudo-parameter must be either ... or the number of parameters that corresponds to the number of arguments in the initial state, __VA_ARGS__.

[data]

Auxiliary data to be passed through the algorithm to the predicate and the operation. This argument may be either a parenthetic expression of any arity or a non-parenthetic expression.

[...]

The initial state of the loop.

Usage - C/C++ Specific 

CHAOS_PP_EXPR(CHAOS_PP_WHILE_AUX(pred, op, data, state))
pred

The predicate that determines whether the loop should continue. This predicate must be a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

pred(s, data, state)
op

The operation used to iterate the state. This operation must be a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

op(s, data, state)
data

Auxiliary data to be passed through the algorithm to the predicate and the operation. This argument may be either a unary parenthetic expression or a non-parenthetic expression.

state

The initial state of the loop.

Remarks

The CHAOS_PP_WHILE_AUX macro itself expands to a deferred expression in terms of CHAOS_PP_NEXT(s). This deferred expression, when invoked by CHAOS_PP_EXPR, expands to the result of repeatedly applying the operation to the state until the predicate expands to 0.

Both the predicate and the operation are invoked with the current recursion state (s), the auxiliary data (data), and the current state (state or __VA_ARGS__). The predicate must expand to either 0, 1, or a deferred expression in terms of CHAOS_PP_NEXT(s) that expands to either 0 or 1. If it expands to 0, the loop terminates, and the algorithm expands to the current state. Otherwise, the operation is applied to the current state to yield a new state or a deferred expression in terms of CHAOS_PP_NEXT(s) that yields a new state, and the loop continues.

The auxiliary data, data, is treated differently depending on whether it is parenthetic. If data is non-parenthetic, it is passed directly to the predicate and the operation as a single argument. In C99, if data is a parenthetic expression of any arity, its sub-elements are passed to the predicate and the operation as individual arguments. In C (prior to C99) and C++, if data is a unary parenthetic expression, its sub-element is passed to the predicate and the operation as a single argument (a parenthetic expression of any other arity is prohibited).

Ignoring the complexity of pred and the op (which are invoked, at most, once per loop iteration), each loop iteration requires only a single recursion step.

This macro deduces the current recursion state and invokes CHAOS_PP_WHILE_AUX_S. If the current recursion state is already known, direct use of CHAOS_PP_WHILE_AUX_S is more efficient.

This macro may not be used if bypass semantics are in effect.

Sample Code

#include <chaos/preprocessor/algorithm/reverse.h>
#include <chaos/preprocessor/array/core.h>
#include <chaos/preprocessor/comparison/not_equal.h>
#include <chaos/preprocessor/control/iif.h>
#include <chaos/preprocessor/control/while_aux.h>
#include <chaos/preprocessor/facilities/as_args.h>
#include <chaos/preprocessor/generics/strip.h>
#include <chaos/preprocessor/list/core.h>
#include <chaos/preprocessor/recursion/expr.h>

#define PRED(s, value, result, list) \
    CHAOS_PP_LIST_IS_CONS(list) \
    /**/
#define OP(s, value, result, list) \
    (2, ( \
        CHAOS_PP_IIF( \
            CHAOS_PP_NOT_EQUAL( \
                value, CHAOS_PP_LIST_HEAD(list) \
            ) \
        )( \
            CHAOS_PP_LIST_CONS( \
                result, CHAOS_PP_LIST_HEAD(list) \
            ), \
            result \
        ), \
        CHAOS_PP_LIST_TAIL(list) \
    )) \
    /**/

#define REMOVE_ALL(value, list) \
    CHAOS_PP_STRIP(CHAOS_PP_REVERSE( \
        (CHAOS_PP_LIST) CHAOS_PP_ARRAY_HEAD( \
            CHAOS_PP_EXPR(CHAOS_PP_WHILE_AUX( \
                PRED CHAOS_PP_AS_ARGS(3), \
                OP CHAOS_PP_AS_ARGS(3), \
                value, (2, (CHAOS_PP_LIST_NIL(), list)) \
            )) \
        ) \
    )) \
    /**/

REMOVE_ALL(3, (3, (1, (2, (3, (1, (3, ...)))))))
// (1, (2, (1, ...)))

CHAOS_PP_WHILE_AUX_ID

The CHAOS_PP_WHILE_AUX_ID macro indirectly expands to CHAOS_PP_WHILE_AUX.

Usage 

CHAOS_PP_WHILE_AUX_ID()

Remarks

This macro effectively allows CHAOS_PP_WHILE_AUX to pass through itself without becoming disabled.

CHAOS_PP_WHILE_AUX_

The CHAOS_PP_WHILE_AUX_ macro expands to a λ expression representing CHAOS_PP_WHILE_AUX.

Usage - C99 Specific 

CHAOS_PP_WHILE_AUX_

Remarks

This macro is a shorthand form of the following λ binding: 

CHAOS_PP_LAMBDA(CHAOS_PP_WHILE_AUX_ID)()
