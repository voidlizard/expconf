CHAOS_PP_HIGHER_ORDER

The CHAOS_PP_HIGHER_ORDER macro is used to simplify the invocations of higher-order constructs.

Usage - C99 Specific 

CHAOS_PP_HIGHER_ORDER()

Usage - C99 Specific 

CHAOS_PP_HIGHER_ORDER(e)(p, ...)
e

The current higher-order entry point.

p

This argument must be empty (i.e. a placemarker).

[...]

The expression to be evaluated in the context of e.

Remarks

This macro is overloaded. The first form of this macro, CHAOS_PP_HIGHER_ORDER(), expands to the current higher-order entry point. The second form expands to __VA_ARGS__.

Neither e or __VA_ARGS__ is allowed to expand on input. In the case of __VA_ARGS__, this pulls the expansion into the context of the macro name resulting from CHAOS_PP_HIGHER_ORDER(e). Each e, which are integers ranging from 1 to CHAOS_PP_LIMIT_HIGHER_ORDER, yields a unique macro name and, therefore, a unique context.

The purpose of this macro is define a small set of entry points that clean up the invocation syntax of higher-order constructs. These invocations usually follow one of the following patterns: 

CHAOS_PP_EXPR(MACRO(...))
CHAOS_PP_EXPR_S(s)(MACRO_S(s, ...))

Given a theoretical higher-order construct MACRO, the CHAOS_PP_HIGHER_ORDER macro can be used to easily map the complex (and more open-ended) invocation syntax to a much simpler one. For example, 

#define AUTO_MACRO CHAOS_PP_CAT(AUTO_MACRO_, CHAOS_PP_HIGHER_ORDER())

#define AUTO_MACRO_1(...) \
    CHAOS_PP_HIGHER_ORDER(1)(, CHAOS_PP_EXPR(MACRO(__VA_ARGS__))) \
    /**/
#define AUTO_MACRO_2(...) \
    CHAOS_PP_HIGHER_ORDER(2)(, CHAOS_PP_EXPR(MACRO(__VA_ARGS__))) \
    /**/

/* ... */

#define AUTO_MACRO_8(...) \
    CHAOS_PP_HIGHER_ORDER(8)(, CHAOS_PP_EXPR(MACRO(__VA_ARGS__))) \
    /**/

This code maps the invocation syntax from CHAOS_PP_EXPR(MACRO(...)) to AUTO_MACRO(...).

The library defines only CHAOS_PP_LIMIT_HIGHER_ORDER higher-order entry points--currently 8. The recursion backend theoretically allows CHAOS_PP_LIMIT_EXPR entry points. However, while many recursion steps may be used, it is unusual to use more the a very small number of entry points.

In order to make a standalone expression such as MACRO(...) operate recursively, MACRO must be an object-like macro that expands to some other macro name that is not currently disabled. That implies, that for eight entry points, eight unique macros are required (such as the AUTO_MACRO_1 through AUTO_MACRO_8 macros above). This is why the library only supports CHAOS_PP_LIMIT_HIGHER_ORDER higher-order entry points--more are rarely needed and increasing their number increases macro replication across all constructs that use it.

If such a mapping is used, a higher-order construct should provide CHAOS_PP_LIMIT_HIGHER_ORDER higher-order entry points because the entry points may be shared across many constructs. Thus, for example, AUTO_MACRO may be invoked only for the first time and still expand to something other than AUTO_MACRO_1 because an invoking macro may have already used the first higher-order entry point. Because of this, a mapping implementation might want to trigger a compile-time assertion if CHAOS_PP_LIMIT_HIGHER_ORDER happens to increase to a number greater than the number of entry-point macros. For example, 

#if CHAOS_PP_LIMIT_HIGHER_ORDER > 8
    #error The AUTO_MACRO implementation needs to be extended.
#endif

Sample Code

#include <chaos/preprocessor/arithmetic/dec.h>
#include <chaos/preprocessor/arithmetic/inc.h>
#include <chaos/preprocessor/cat.h>
#include <chaos/preprocessor/config.h>
#include <chaos/preprocessor/control/when.h>
#include <chaos/preprocessor/punctuation/comma_if.h>
#include <chaos/preprocessor/recursion/basic.h>
#include <chaos/preprocessor/recursion/expr.h>
#include <chaos/preprocessor/recursion/higher_order.h>

#define REP(count, macro, data) \
    REP_S(CHAOS_PP_STATE(), count, macro, data) \
    /**/
#define REP_S(s, count, macro, data) \
    REP_I(CHAOS_PP_OBSTRUCT(), CHAOS_PP_NEXT(s), count, macro, data) \
    /**/
#define REP_INDIRECT() REP_I
#define REP_I(_, s, count, macro, data) \
    CHAOS_PP_WHEN _(count)( \
        CHAOS_PP_EXPR_S _(s)(REP_INDIRECT _()( \
            CHAOS_PP_OBSTRUCT _(), CHAOS_PP_NEXT(s), \
            CHAOS_PP_DEC(count), macro, data \
        )) \
        macro _(s, CHAOS_PP_DEC(count), data) \
    ) \
    /**/

#if CHAOS_PP_VARIADICS

#if CHAOS_PP_LIMIT_HIGHER_ORDER > 8
    #error The AUTO_REP implementation needs to be extended.
#endif

#define AUTO_REP CHAOS_PP_CAT(AUTO_REP_, CHAOS_PP_HIGHER_ORDER())

#define AUTO_REP_1(...) CHAOS_PP_HIGHER_ORDER(1)(, CHAOS_PP_EXPR(REP(__VA_ARGS__)))
#define AUTO_REP_2(...) CHAOS_PP_HIGHER_ORDER(2)(, CHAOS_PP_EXPR(REP(__VA_ARGS__)))
#define AUTO_REP_3(...) CHAOS_PP_HIGHER_ORDER(3)(, CHAOS_PP_EXPR(REP(__VA_ARGS__)))
#define AUTO_REP_4(...) CHAOS_PP_HIGHER_ORDER(4)(, CHAOS_PP_EXPR(REP(__VA_ARGS__)))
#define AUTO_REP_5(...) CHAOS_PP_HIGHER_ORDER(5)(, CHAOS_PP_EXPR(REP(__VA_ARGS__)))
#define AUTO_REP_6(...) CHAOS_PP_HIGHER_ORDER(6)(, CHAOS_PP_EXPR(REP(__VA_ARGS__)))
#define AUTO_REP_7(...) CHAOS_PP_HIGHER_ORDER(7)(, CHAOS_PP_EXPR(REP(__VA_ARGS__)))
#define AUTO_REP_8(...) CHAOS_PP_HIGHER_ORDER(8)(, CHAOS_PP_EXPR(REP(__VA_ARGS__)))

#define TEXT(s, n, text) CHAOS_PP_COMMA_IF(n) text
#define TTP(s, n, id) \
    CHAOS_PP_COMMA_IF(n) \
    template<AUTO_REP(CHAOS_PP_INC(n), TEXT, class)> class id ## n \
    /**/

AUTO_REP(3, TTP, T)
// template<class> class T0,
// template<class, class> class T1,
// template<class, class, class> class T2

#endif

CHAOS_PP_HIGHER_ORDER_ID

The CHAOS_PP_HIGHER_ORDER_ID macro indirectly expands to CHAOS_PP_HIGHER_ORDER.

Usage - C99 Specific 

CHAOS_PP_HIGHER_ORDER_ID()

Remarks

This macro exists only for uniformity with other macros that require indirect invocation.

CHAOS_PP_HIGHER_ORDER_

The CHAOS_PP_HIGHER_ORDER_ macro expands to a λ expression representing CHAOS_PP_HIGHER_ORDER.

Usage - C99 Specific 

CHAOS_PP_HIGHER_ORDER_

Remarks

This macro is a shorthand form of the following λ binding: 

CHAOS_PP_LAMBDA(CHAOS_PP_HIGHER_ORDER)
