The seq/transform.h header defines an algorithm that transforms the elements of a sequence with a user-specified operation.

Usage

#include <chaos/preprocessor/seq/transform.h>

Contents

  • CHAOS_PP_SEQ_TRANSFORM

CHAOS_PP_SEQ_TRANSFORM

The CHAOS_PP_SEQ_TRANSFORM macro transforms each element of its second argument with the operation specified by its first argument.

Usage - C99 Specific 

CHAOS_PP_EXPR(CHAOS_PP_SEQ_TRANSFORM(op, seq))
op

The operation used to transform each element of seq. This operation must be either a λ expression or a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

op(s, elem†)

† If seq contains elements that have varying arities, elem must be .... Otherwise, elem must be the number of parameters that corresponds to the arity of each element in seq.

seq

The sequence whose elements are to be transformed.

Usage - C99 Specific 

CHAOS_PP_EXPR(CHAOS_PP_SEQ_TRANSFORM(op, seq, ...))
op

The operation used to transform each element of seq. This operation must be either a λ expression or a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

op(s, elem†, data‡)

† The elem pseudo-parameter must be the number of parameters that corresponds to the arity of each element in seq.

‡ The data pseudo-parameter must be either ... or the number of parameters that corresponds to the number of arguments in the auxiliary data, __VA_ARGS__.

seq

The sequence whose elements are to be transformed.

[...]

Auxiliary data to be passed through the algorithm to the operation.

Usage - C/C++ Specific 

CHAOS_PP_EXPR(CHAOS_PP_TRANSFORM(op, seq, data))
op

The operation used to transform each element of seq. This operation must be a macro name that refers to a macro with a signature compatible with the following pseudo-signature: 

op(s, elem, data)
seq

The sequence whose elements are to be transformed.

data

Auxiliary data to be passed through the algorithm to the operation.

Remarks

The CHAOS_PP_SEQ_TRANSFORM macro itself expands to a deferred expression in terms of CHAOS_PP_NEXT(s). This deferred expression, when invoked by CHAOS_PP_EXPR, expands to a sequence that contains the transformed elements of seq.

The operation is invoked with the current recursion state (s), a particular element (elem), and, if it exists, the auxiliary data (data or __VA_ARGS__). This invocation must expand to an element that replaces the element for which it is invoked or a deferred expression in terms of CHAOS_PP_NEXT(s) that expands to such an element. Expansion to a deferred expression allows the operation to recurse on itself if necessary.

In C99, it is possible to omit the auxiliary data argument altogether. This is different than passing "nothing" as the auxiliary data. 

CHAOS_PP_EXPR(
    CHAOS_PP_SEQ_TRANSFORM(OP, (a)(b)(c),) // #1
    CHAOS_PP_SEQ_TRANSFORM(OP, (a)(b)(c))  // #2
)

The first line passes "nothing" as the auxiliary data, which causes OP to be invoked as (e.g.)... 

OP(s, a,)

The second line omits the auxiliary data, which causes OP to be invoked as (e.g.)... 

OP(s, a)

The fundamental difference is that, in the first line, the auxiliary data is empty (which might have a valid meaning as data), but, in the second line, there is no auxiliary data. Hence, the algorithm is overloaded on a true-optional parameter.

Invocations of the operation are trampolined such that each invocation is invoked with CHAOS_PP_NEXT(s), where s is the original recursion state at the beginning of the algorithm (i.e. CHAOS_PP_STATE()). This maximizes the utility of the recursion backend.

This macro uses sequential iteration. As a result, the efficiency of the algorithm's use of the recursion backend, described in terms of the number of recursion steps required to complete the algorithm and ignoring the complexity of op, is a constant Θ(1). As such, the algorithm is theoretically capable of processing sequences containing an unrestricted number of elements.

This macro deduces the current recursion state and invokes CHAOS_PP_SEQ_TRANSFORM_S. If the current recursion state is already known, direct use of CHAOS_PP_SEQ_TRANSFORM_S is more efficient.

This macro may not be used if bypass semantics are in effect.

Sample Code

#include <chaos/preprocessor/config.h>
#include <chaos/preprocessor/control/if.h>
#include <chaos/preprocessor/lambda/ops.h>
#include <chaos/preprocessor/recursion/expr.h>
#include <chaos/preprocessor/seq/transform.h>

#define OP(s, elem, data) CHAOS_PP_IF(elem)(elem, data)

CHAOS_PP_EXPR(CHAOS_PP_SEQ_TRANSFORM(
    OP,
    (1)(0)(0)(5),
    ~
))
// (1)(~)(~)(5)

#if CHAOS_PP_VARIADICS

??=include CHAOS_PP_PLACEHOLDERS(1)

CHAOS_PP_EXPR(CHAOS_PP_SEQ_TRANSFORM(
    CHAOS_PP_IF_(_1)(_1, ~),
    (1)(0)(0)(5)
))
// (1)(~)(~)(5)

??=include CHAOS_PP_PLACEHOLDERS(0)

#endif

CHAOS_PP_SEQ_TRANSFORM_ID

The CHAOS_PP_SEQ_TRANSFORM_ID macro indirectly expands to CHAOS_PP_SEQ_TRANSFORM.

Usage 

CHAOS_PP_SEQ_TRANSFORM_ID()

Remarks

This macro effectively allows CHAOS_PP_SEQ_TRANSFORM to pass through itself without becoming disabled.

CHAOS_PP_SEQ_TRANSFORM_

The CHAOS_PP_SEQ_TRANSFORM_ macro expands to a λ expression representing CHAOS_PP_SEQ_TRANSFORM.

Usage - C99 Specific 

CHAOS_PP_SEQ_TRANSFORM_

Remarks

This macro is a shorthand form of the following λ binding: 

CHAOS_PP_LAMBDA(CHAOS_PP_SEQ_TRANSFORM_ID)()
